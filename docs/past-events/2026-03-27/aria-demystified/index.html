What is ARIA?

  • Complementary w3 standard for HTML and SVG
  • Expose meaning of markup in a machine-readable way
  • Web semantics → OS a11y-API

Who consumes a11y APIs?

  • Assistive Technologies (AT)
    • Braille displays
    • Screen magnifiers
    • Text-to-speech software
    • Screen readers
    • ... and more

Basic VoiceOver Usage

  • VO-Modifier: Cmd-Opt or CapsLock
  • Navigate
    • Move VO-cursor around
    • Find next/previous element of a specific type (heading, link, bold, table, etc.)
  • Interact
    • Read content, sentence, paragraph, etc. under VO-cursor
    • Start/stop item interaction/inspection
    • Click, select, drag etc. focused item
  • Orient
    • summarize and filter (Rotor)

Using ARIA: Landmarks

landmark role
A perceivable section containing content that is relevant to a specific, author-specified purpose and sufficiently important that users will likely want to be able to navigate to the section easily [...]
- WAI-ARIA 1.2

Using ARIA: Landmarks

  • Important sections on web thing (author-defined)
  • Generally recognizable on different web things (not domain-specific)
  • Allow user to create a coarse mental model of the thing
  • Can be used for orientation/navigation

Using ARIA: Landmarks

Specified with role attribute OR by using appropriate tag

Using ARIA: Landmarks

Takeaways

  • Identify landmarks in your web thing and use the appropriate tags/roles
  • Prefer native HTML elements over role attribute
  • Use landmarks sparingly, too many defeat their purpose
  • If you have multiple landmarks of the same type, name them appropriately

Using ARIA: Naming Things

  • Proper naming is crucial to make a web thing accessible for screen readers
  • Missing/incorrect naming is akin to CSS * { display: none; }
  • Browser can try fallbacks to name things, but those are not always useful

Using ARIA: Naming Things

How to set an accessible name

Mechanism Example
Element's text content 
<h1>Hello Winti Web</h1>
label, figcaption, caption, legend tags 
<label>Hello Winti Web <input /></label>
aria-label attribute 
<div aria-label="Hello Winti Web">
aria-labelledby attribute 
<div aria-labelledby="some-id other-id">
<div id="some-id" aria-label="Hello">
<div id="other-id" aria-label="Winti Web">
alt, title attributes 
<img alt="Hello Winti Web">
input placeholder (not recommended) 
<input placeholder="Hello Winti Web"/>

Using ARIA: Naming Things

Takeaways

  • If possible, design your thing so visible text == accessible name
  • Give elements succinct names, that convey purpose and intent
  • Don't include role in name (e.g. use "product" for search, not "product search")
  • For longer (concatenated) labels, put important part at front
  • Don't forget to translate names
  • Test with a screen reader

Using ARIA: Reducing Noise and Confusion

  • Sound of your web thing may be overly verbose
  • Relying too heavily on generic tags may confuse user
  • Using semantic tags in unintended ways, may confuse user
  • Reducing superfluous information, using meaningful tags and hiding decorative elements from AT improves legibility

Using ARIA: Reducing Noise and Confusion

Takeaways

  • Be careful when hiding text by opacity/transparency, font-size, or positioning offscreen
  • Use tags with appropriate semantics whenever possible
  • Hide decorative items from AT with aria-hidden

Using ARIA: Live Regions

  • Elements that don't have focus, may have their content changed
  • AT needs to be informed about change, without having to move focus there

Recap

  • Identify and mark essential landmarks appropriately, name them if necessary
  • Ensure everything that is not purely decorative has an accessible name
  • Don't forget to translate accessible names
  • Use semantic tags or roles to declare intent of an element
  • Reduce noise by hiding decorative elements
  • Mark dynamically changing regions with aria-live or appropriate roles
  • ACTUALLY TEST WITH A SCREEN READER
